# 设计决策

## 1. 目标

记录项目里的关键技术选择和取舍，方便后续维护，解释“为什么这样设计”。

## 2. 核心选择

### 2.1 使用 YOLO 做人体检测

为什么这么做：

- 和项目目标一致
- 本地推理可行
- 输出框和置信度，便于后续跟踪

备选方案：

- 传统目标检测器
- 更大的检测模型
- 直接做人脸检测

为什么没选：

- 传统方案在复杂家庭场景里泛化弱
- 更大的模型推理更慢，不适合当前 Demo
- 直接做人脸检测会让前置条件更苛刻，不利于先把“人在哪”这件事跑通，先找“person”再分割人脸会好一点，不像门禁场景中画面一般都是人脸，机器人场景中更多的是整个person进入画面。

代价：

- 模型越大，延迟越高
- 精度和速度需要平衡

### 2.2 使用 DeepSORT 做多目标跟踪

为什么这么做：

- 和项目目标一致
- 适合短期轨迹连续性
- 能把临时轨迹 ID 和长期身份分开
- 对 Demo 来说足够直观

备选方案：

- ByteTrack
- 只靠检测框做帧间关联

为什么没选：

- DeepSORT 已经能满足当前稳定跟踪和身份分离的需要

代价：

- 遮挡或离开后，track ID 可能变化
- 跟踪本身不能解决长期身份问题

### 2.3 身份层单独设计

为什么这么做：

- 轨迹和身份不是一回事
- 家庭成员需要长期稳定身份
- 身份恢复不能依赖 track ID

备选方案：

- 直接把 track ID 当身份
- 在跟踪层内顺带做身份绑定

为什么没选：

- track ID 会在离开、遮挡、重连后变化，无法实现记忆功能
- 把身份塞进跟踪层会让两类问题耦合，后续很难维护，跟踪层和身份层分离测试起来也更加方便
- 家庭成员的“是谁”必须独立于“这一帧的轨迹编号”

当前技术路线：

- `InsightFace` 负责本地人脸特征提取
- `FAISS` 负责向量检索与身份匹配
- 本地登记库负责保存长期身份样本
- `ReID` 负责轨迹消失后的身份恢复

这样写的好处是，视觉层负责“看见并跟住”，身份层负责“认出来并记住”，边界比较清晰。

这也是为什么项目里会区分：

- `track_id`：临时
- `identity_id`：长期

### 2.4 使用本地登记

为什么这么做：

- 适合实际应用场景 类似门禁中的人脸识别
- 可以演示“先登记、再识别”
- 不依赖云端服务

备选方案：

- 云端身份库
- 临时游客模式
- 完全不做登记，直接开放识别

为什么没选：

- 本地注册身份在这个项目开发过程中更加方便，云端身份库还要考虑云服务器运行和维护，网络开销等问题
- 游客模式不符合“家庭成员长期识别”的主线


代价：

- 需要手工初始化登记数据

### 2.5 语音采用本地唤醒 + ASR + TTS

为什么这么做：

- 能独立演示语音链路
- 便于和视觉身份联动
- 方便后续继续扩展，比如接入 LLM 形成更完整的对话能力；同时也可以基于身份登记形成不同用户的偏好记忆，例如爷爷偏养生话题，小朋友偏动漫话题。

备选方案：

- 直接云端语音服务
- 只做文本交互，不做语音
- 先做大模型对话，再补语音链路
- Piper 本地 TTS

为什么没选：

- 云端方案受网络影响
- 只做文本交互实现起来确实比较简单，但是不符合家庭机器人 Demo 的体验，接入语音服务人机交互更好也便于后续扩展
- Piper 是很好的离线 TTS 方案，但还需要额外管理语音模型和执行文件；当前阶段先保留 edge-tts 提升自然度，同时用 pyttsx3 做本地兜底，更适合 Demo 交付节奏

当前策略：

- 唤醒词只负责触发
- 回应由当前视觉身份决定，例如识别到画面中是爸爸，收到唤醒词之后就返回爸爸对应的问候语。
- TTS 负责输出

补充说明：

- 当前工程实现是 `edge-tts` 优先、`pyttsx3` 兜底，因此实际表现为“在线优先 + 本地回退”
- 如果后续想把 TTS 完全改成本地部署，Piper 是优先考虑的候选方案


### 2.6 录制视频音频单独提取

为什么这么做：

- 便于按时间轴对齐唤醒事件
- 录制视频测试和实时摄像头测试可复用同一引擎

备选方案：

- 只看视频帧，不处理音频
- 只做实时摄像头，不支持录制视频

为什么没选：

- 不处理音频就无法验证唤醒词链路
- 只做实时摄像头会让某些难以复现，比如这个项目中要测试一家五口，还是需要视频源来进行测试

代价：

- 需要额外的音频提取步骤
- 对齐精度受视频 FPS 和音频质量影响
- 要解决音视频同步等问题，不过考虑到实际应用场景应该都是机器人摄像头捕获画面，视频分析仅作为测试
### 2.7 多人场景发起者推测采用 mock

为什么这么做：

- 单个普通麦克风很难在多人同场时精确判断说话人，比如在一个画面中有一家五口，有人触发了唤醒词，但是我不知道是谁触发的，不能触发具体交互
- 这个阶段更适合先把演示流程跑通
- 真正的说话人识别会引入更复杂的硬件和算法

备选方案：

- 麦克风阵列
- 声源定位
- 活跃说话人检测
- 音视频对齐
- 唇动辅助判断

为什么没选：

- 这些方案比当前 Demo 阶段复杂程度高不少
- 需要额外硬件或更复杂的数据链路(缺失麦克风阵列等)
- 当前目标是先把“视觉身份 + 语音触发 + 问候”跑通

当前 mock 方案：

- 根据视觉身份状态推测发起者
- 只在识别到 2 个及以上已登记身份时启用，单人场景不进入推测流程
- 优先选择更稳定、更靠近画面中心、近期更活跃的已识别成员
- 多人同时在场时，明确标注为“推测结果”

正式方案可在后续升级为上面这些能力的组合。


## 3. 已知局限

- 启动阶段第一句唤醒词可能漏识别，尤其在视频识别场景下，可能和音视频同步有关，已记录在 `blockers.md` 中
- 中文 ASR 的准确率受模型和音频质量影响，所以本次测试中的唤醒词尽量简单
- 视觉身份稳定性受登记样本质量影响（比如光线 遮挡等）人脸分割、识别的算法也可以设计的更精细一点
- 在本次demo中对硬件的算力（gpu）有一定要求，如果要边缘部署的话还需要做更多模型性能的优化
## 4. 未来发展

- 关于这个项目的落地和扩展
- 如果未来在开发板上部署这个项目，需要再考虑模型量化、部署优化以及更工程化的实现方式；目前 demo 已经把核心逻辑链路跑通，后续无论接入 LLM 还是扩展视觉与 TTS，都有比较清晰的基础。
- 这次没有单独抽出 `harness` 层，主要是为了先把主链路跑通；但从复盘来看，后续如果继续扩展，独立的运行/测试/复现入口会更利于排查视频、音频和 GUI 的联动问题。
