# AI 协作笔记

这份文档记录我和 Codex 协作时，真正有复用价值的对话模式、决策方法和踩坑经验。  

## 1. 协作原则

### 1.1 一次只做一个任务

- 先看 `AGENTS.md`
- 再看 `SPEC.md`
- 再看 `TASKS.md`
- 永远只处理当前第一条未完成任务
- 不提前做未来任务

这个规则非常重要。  
它能避免“AI 为了实现需求代码越写越多，但任务状态越来越混乱，最后偏离任务”。

### 1.2 先定边界，再动手

在开始实现前，先确认：

- 目标是什么
- 不做什么
- 这一步完成后要怎么验收
- 是否会影响后续模块

如果边界不清，AI 很容易自己补假设，最后偏题。

### 1.3 每次只改一小段

适合这个项目的节奏是：

- 一个模块
- 一次小改动
- 一轮测试
- 再继续

这样最容易定位问题，也方便人工审查。

### 1.4 移动端也可以做复核

在本次开发过程中，如果有不方便盯着电脑的时候，比如（拍毕业照、和导师谈论文😓），可以用手机 GPT App / Codex 移动版做这些事：

- 看当前任务进度
- 复核某一步是否偏题
- 检查文档里有没有漏掉的约束
- 快速确认下一步是不是应该继续做

它更适合做“轻量审查”和“方向确认”，不适合替代桌面端的实际改代码和跑测试。

## 2. 有效对话方式

### 2.1 把需求写成可执行目标

比起“帮我优化一下”，更有效的是：

- “把人脸识别阈值降到 0.65”
- “把登记数据迁到项目根目录”
- “打印各个模型的推理时间和端到端的总时间”

比起“帮我完成视觉模块、人脸识别模块”，更有效的是：
- “使用 YOLOv8 进行 person 框标定，DeepSORT 进行轨迹跟踪，InsightFace 进行人脸识别”
- “各个模块的技术选型查看design.md”
目标越具体，改动越稳定，不容易出错。

### 2.2 明确验收标准

对 Codex 最有用的验收方式是：

- 代码改了什么
- 测试跑了什么
- 结果是否通过
- 还有没有残余风险

这比只说“看看对不对”要有效得多。

AI 在对某个模块做测试的时候，会先用模拟数据。比如测试 ASR 模块时，可能会用模拟音频；如果模拟数据不够真实，就需要再找实际数据自己验证。

### 2.3 让 AI 解释取舍

很多时候不是“做不做”，而是“为什么这样做”。

比较有价值的问题包括：

- 为什么 Track ID 不能当身份
- 为什么唤醒词只负责触发
- 为什么 YOLO + DeepSORT 不够
- 为什么不能弃用 DeepSORT，直接使用 YOLO + InsightFace 人脸检测

这种问题能让系统设计真正落地，而不是停留在功能堆砌。比如“为什么不能弃用 DeepSORT，直接使用 YOLO + InsightFace 人脸检测”：因为 InsightFace 的推理开销更大，DeepSORT 可以先稳定轨迹，再用轨迹去做身份映射。

## 3. 这次项目里沉淀出的关键方法论

### 3.1 Track ID 和 Identity 必须分离

- Track ID 是临时的
- Identity 是长期的
- 人离开再回来，Track ID 可以变，Identity 不应该变

这是整套系统最核心的设计原则之一。

### 3.2 检测、跟踪、身份识别要分层

推荐理解方式：

1. YOLO 负责找人（person框的标注）
2. DeepSORT 负责短期连续跟踪
3. 身份层负责长期身份记忆

如果把三件事混成一件事，后面一定会乱。

### 3.3 本地模型和数据要放在项目目录

为了方便复现，最终统一成：

- `models/`
- `data/enrollment/`

不要让核心模型悄悄跑到用户目录缓存里。  
否则别人拿到项目后，复现路径会碎掉。

### 3.4 性能优化要先看“真正的瓶颈”

这次实践里最重要的一点是：

- 不是所有“变慢”都来自模型本身
- 有时候是文本渲染、音频监听、线程调度造成的

所以要先看日志通过打印出的时间，再判断是：

- 视觉推理慢
- 语音等待慢
- GUI 绘制慢
- 还是音视频同步慢

比如说，一次改动后推理时间变长，可能是 GUI 上的文本框绘制拖了后腿，不一定是模型推理速度变慢了。

### 3.5 先保留阻塞点，再继续推进

遇到暂时解决不了的问题，不要硬绕。

更好的做法是：

- 先记录到 `docs/blockers.md`
- 明确现象、影响和临时规避方式
- 再继续完成主线任务

这样不会因为一个细节卡住整个项目。

## 4. 这次协作里最有价值的表达模板

### 4.1 “先确认，再实现”

适用于：

- 路径变更
- 阈值调整
- 模型位置迁移
- GUI 交互调整

### 4.2 “先测试，再扩展”

适用于：

- 人脸识别
- 语音唤醒
- 视频音频联动
- 实时 GUI

先确认核心链路成立，再加功能，单个模块的功能测试通过再扩展其他部分。

### 4.3 “先记录，再继续”

适用于：

- 第一帧漏识别
- 某些场景鲁棒性不足
- 依赖版本不固定
- 某些模型还没下载

先把问题写进文档，项目节奏会稳很多。在增加需求时，最好先把 `AGENTS.md`、`TASKS.md` 等设计文档补好，再让 AI 按这些文档开发。

## 5. 可以复用的检查清单

每次让 AI 改项目时，可以快速问这几句：

1. 当前第一条任务是什么？
2. 这一步的边界是什么？
3. 改完要怎么验证？
4. 是否影响已有模块？
5. 是否需要同步更新文档？
6. 是否集成到 GUI（主函数）中？


## 6. 最后总结

这次协作最重要的收获不是某个具体模块，而是这套工作节奏：

- 任务驱动
- 一次一件事
- 先验收再扩展
- 文档和代码一起推进
- 所有核心资源尽量本地化、可复现
